# 触发与项目配置指南

Workspace Health Review 是显式触发或周期运行的治理审计，不应该在每次实现任务中自动执行。

## 推荐触发方式

对于持续活跃的仓库，可以每周或按团队约定的周期运行一次。以下情况也建议启用：

- 到达重要里程碑；
- OpenSpec change、ADR、项目指令、Skill 或其他持久化记录明显积累；
- 准备归档、发版就绪检查、成员入场、所有权交接，或恢复长期中断的工作；
- 规则疑似过期、重复、冲突，或者工作区已经难以恢复。

日常闲聊、普通只读问答、简单命令输出，以及没有暴露工作区级健康信号的常规实现任务，默认不启用。

默认采用 `Analyze First`。检查只生成证据和建议；未经人工明确批准，不得归档、删除、重写或初始化治理文件。

## 让 AI 帮你配置项目

可以把下面的提示词交给 Agent：

```text
请在 AGENTS.md 中为 $workspace-health-review 添加一条简洁的项目级触发规则。
保留所有已有指令，不要重复已有的审查规则。对于活跃仓库，建议显式或周期运行；
在重要里程碑、OpenSpec 或持久化记录积累、准备归档或发版就绪检查、成员入场、
所有权交接、长期中断恢复时启用。不要在每次实现任务中自动执行。
默认采用 Analyze First，修改治理文件前必须获得人工批准。写入前先向我展示拟新增的区块。
```

确认 Agent 提议的内容后再允许写入。Agent 应追加或合并最小规则，而不是覆盖原有 `AGENTS.md`。

## 手动添加

可以复制并按项目情况调整：

```markdown
## Workspace Health Review

- 活跃仓库应周期使用 `$workspace-health-review`，并在重要里程碑或持久化记录、规则、Skill 明显积累时检查。
- 准备归档、发版就绪检查、成员入场、所有权交接和长期中断恢复时也建议使用。
- 不在每次实现任务或简单只读请求中自动执行。
- 默认 Analyze First；未经明确批准，不修改治理文件。
```

检查周期应由项目自己决定。每周一次适合活跃工作区，但不是所有项目必须遵守的固定要求。

Claude Code、Cursor、Gemini CLI 或其他 Agent 可以把相同规则调整到各自的项目指令文件中。
